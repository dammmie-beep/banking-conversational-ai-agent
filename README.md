# Bank Conversational AI Agent

A fully offline conversational AI agent for Bank customer service. Built with Flask, llama.cpp, and a small embedding model — no internet connection required after initial setup.

---

## Features

- **Product Queries** — Answer questions about bank products (loans, savings, investments, debit cards) via RAG (FAISS semantic search)
- **Account Queries** — Retrieve customer account details and recent transaction history
- **ATM Card Blocking** — Multi-turn card blocking flow with support for multiple cards per account; state resets when the session ends
- **Sliding Window Memory** — Maintains conversation context across multiple turns per session (10-turn window)
- **Fully Offline** — Runs entirely on local hardware after one-time model download

---

## Tech Stack

| Component | Technology |
|---|---|
| API Framework | Flask |
| LLM | Qwen2.5-3B-Instruct (GGUF Q4_K_M) via llama.cpp |
| Embedding Model | sentence-transformers/all-MiniLM-L6-v2 |
| Vector Store | FAISS (CPU) |
| Database | SQLite (generated from Excel via `sqlite.py`) |
| Runtime | Python 3.8+ |

## Installation

### Step 1 — Clone the repository
```bash
git clone https://github.com/dammmie-beep/banking-conversational-ai-agent.git
cd bank_agent
```

### Step 2 — Create and activate a virtual environment
```bash
python -m venv venv

# Windows
venv\Scripts\activate

# Mac/Linux
source venv/bin/activate
```

### Step 3 — Install dependencies
```bash
pip install -r requirements.txt

# Linux/Mac — build llama-cpp-python with OpenBLAS for faster CPU inference
CMAKE_ARGS="-DLLAMA_BLAS=ON -DLLAMA_BLAS_VENDOR=OpenBLAS" \
  pip install llama-cpp-python --force-reinstall --no-cache-dir
```

### Step 4 — Download the LLM model (one time, requires internet)
```bash
huggingface-cli download Qwen/Qwen2.5-3B-Instruct-GGUF \
  qwen2.5-3b-instruct-q4_k_m.gguf \
  --local-dir ./models
```

### Step 5 — Cache the embedding model (one time, requires internet)
```bash
python -c "
from sentence_transformers import SentenceTransformer
model = SentenceTransformer('sentence-transformers/all-MiniLM-L6-v2')
model.save('./models/minilm')
print('Embedding model cached.')
"
```

### Step 6 — Generate the database
Place your Excel data file in `data/` then run:
```bash
python sqlite.py
```
This creates `data/banking.db` from the Excel sheets (Customers, Transactions, Cards).

### Step 7 — Start the server
```bash
python run.py
```

Server starts at `http://0.0.0.0:5000`

### Health Check
```
GET /api/health

```

### Chat
```
POST /api/chat
Content-Type: application/json
```
```json
{ "session_id": "user-session-1", "message": "What is my account balance?" }
```
```json
{ "session_id": "user-session-1", "response": "..." }
```

> Use the same `session_id` across requests to maintain conversation context. Each new `session_id` starts a clean session with no memory of previous interactions.

### View Conversation History
```
GET /api/session/{session_id}/history
```

### Clear Session
```
DELETE /api/session/{session_id}/clear
```

## How It Works

```
POST /api/chat
  → routes.py              — HTTP validation, session_id generation
  → agent.run_agent()      — Core orchestration
      → route_with_llm()       — LLM classifies intent (pure LLM, no keyword fallback)
      → handle_block_card()    — Multi-turn card blocking (session-scoped state)
      → handle_account_query() — Account/transaction lookup + LLM summary
      → handle_product_query() — FAISS semantic search (RAG) + LLM answer
      → handle_general()       — Free-form conversational response
  → memory.py              — Per-session deque (10-turn sliding window)
```

### Card Blocking Flow
```
1. User requests to block a card
2. Agent asks for account number (if not already in session)
3. Agent fetches linked cards
4. Single active card → asks for confirmation → blocks
5. Multiple active cards → lists them → user selects → confirms → blocks
6. Blocked status is session-scoped — starts fresh on a new session
```

## Configuration

All settings are in `config.py`:

```python
N_CTX = 2048       # LLM context window
N_THREADS = 4      # CPU threads (set to your CPU core count)
MAX_TOKENS = 150   # Maximum response length
TEMPERATURE = 0.3  # Lower = more deterministic, slightly faster
N_BATCH = 512      # Prompt evaluation batch size (larger = faster prefill)
WINDOW_SIZE = 10   # Conversation turns to retain in memory
```

## Known Limitations

- **Response time**: 30–90 seconds per message on CPU-only machines; GPU support via llama.cpp can reduce this significantly
- **Qwen2.5-3B** is a small model — occasional imprecision on complex reasoning
- Sessions are **in-memory only** — all session state (including card blocking) resets on server restart
